# Files → Attachments — Design

## Context

`FilesView.tsx` currently renders `groupFilesByAgent(state.agents)` — the fictional roster's
`Agent.files: AgentFile[]`, grouped under each agent's name and hue, matching the original
"honest placeholder" design from the initial scaffold. This was the last unmigrated Phase 3
target (Terminal/Dashboard widgets, the Agents view, the Orchestration Grid, and Analytics'
`AgentBreakdownCard` were already ported to real dispatch data in slices 1–4).

Before designing this slice, real-agent-data feasibility for Files was investigated directly:
`~/.claude/projects/**/*.jsonl` was grepped fleet-wide for `isSidechain:true` (how Claude Code
would mark a subagent's own tool calls if recorded inline) — zero hits anywhere. A real
historical `Agent` dispatch's completion event was also traced to its `<output-file>` — that
file holds only the subagent's final text result, is ephemeral, and does not log tool calls.
**Conclusion: Claude Code does not persist a subagent's internal tool-call history (which files
it touched) anywhere on disk.** Unlike slices 1–4, there is no missing plumbing to build here —
the underlying data required for an honest "real files touched" view does not exist. This was
confirmed with the user, who chose to pivot the slice's purpose entirely rather than force a
relabeling of fictional data (the way Analytics reframed its card in slice 4).

**`Agent.files`/`AgentFile` stay exactly as they are** — grep confirms they're still actively
read by Chat's `localResponder.ts` (offline fallback copy) and `systemPrompt.ts` (fed into the
live Claude system prompt as `filesTouched` for a fictional agent persona). This slice only
removes `FilesView.tsx`'s *own* consumption of that data; nothing about the fictional roster or
Chat is touched. `groupFilesByAgent` (`src/components/files/filesMath.ts`) has exactly one
caller — `FilesView.tsx` — confirmed by grep, so it becomes dead code once this slice's
implementation task rewires that component, and is removed in the same task (matches this
project's "remove dead code only once genuinely safe" precedent).

## Goal

Turn the Files tab into a real, standalone local file-attachment library: add a file via a
native OS picker, see it listed (image files show an inline thumbnail; other types show a
generic badge), open any file in its OS default application, delete a file you no longer need.
Intended use: dropping in a screenshot or log file while troubleshooting something with the
user, without needing a second app open.

## Non-goals

- **No Chat integration.** Attachments are not referenceable from any chat channel in this
  slice — a chat persona cannot "see" that a file was attached. This is a real, deliberate scope
  cut discussed with the user (matches this project's precedent of splitting large features
  across slices, e.g. Chat's action-execution pipeline was its own later phase, 2b, after the
  base chat view shipped in 2a). Worth a dedicated future slice if wanted.
- **No drag-and-drop.** Add is picker-only (`dialog.showOpenDialog`). Electron 32+ removed
  direct `file.path` access from HTML5 drag events, so drag-and-drop needs the newer
  `webUtils.getPathForFile` API — an extra integration point deliberately deferred.
- **No filesystem-watching.** The list refreshes on view mount and immediately after an
  in-app add/remove — not if a file is dropped into `~/.aether-os/attachments/` from outside the
  app (e.g. Windows Explorer) while the view is open. Stated explicitly rather than discovered
  missing later.
- **No live screen capture.** Attaching a screenshot means attaching a file you already saved
  (e.g. via the OS's own screenshot tool), not capturing the screen from inside this app. Would
  need Electron's `desktopCapturer` API — a materially bigger add, out of scope here.
- **No file-type or size restriction, no validation.** Any file the OS picker allows can be
  attached; no size cap, no extension allowlist beyond the separate image-detection used for
  thumbnails (see Architecture). Matches this app's general pattern of not adding defensive
  validation with no concrete failure mode driving it.
- **No association with any agent, dispatch, or project.** A flat, ungrouped list — not
  "grouped by who requested it" or similar; there is no real concept to group by (attachments
  are added by the user, not generated by any agent).

## Architecture

Reuses the existing `src/components/files/` directory. **Correction from an initial draft of
this doc:** `ViewDef` has no separate display-label field — `TopBar.tsx` renders each view's
`id` directly as the tab's visible text (confirmed by reading `TopBar.tsx`), so relabeling the
tab to "Attachments" means changing `viewRegistry.ts`'s entry from `id: 'Files'` to
`id: 'Attachments'`, not just a cosmetic label swap. Grepped every `.ts`/`.tsx` file for the
literal string `'Files'` first to size the blast radius: exactly two hits, the registry entry
itself and `viewRegistry.test.ts`'s two assertions — nothing else in the app dispatches
`SET_ACTIVE_TAB('Files')`, so the rename is contained to those two files. A stale `'Files'`
string in a prior session's persisted `activeTab` simply won't match any current view (no tab
highlighted, no crash) — an acceptable, self-correcting edge case for a single-user local app,
not worth defensive migration code.

### New storage location

`~/.aether-os/attachments/` — mirrors TokenMonitor's `~/.claude-token-tracker/` convention
(consistent with this project's stated direction of becoming the "evolution of TokenMonitor").
Created on first use if it doesn't exist.

### `electron/attachmentsStore.ts` (new)

Mirrors `liveAgentTracker.ts`'s role as the electron-side module owning one piece of real state:

- `list(): Promise<{ name: string; size: number; mtimeMs: number }[]>` — reads the directory,
  newest-first.
- `add(): Promise<string[]>` — opens `dialog.showOpenDialog({ properties: ['openFile',
  'multiSelections'] })`; for each chosen path, copies the file into the attachments folder,
  resolving name collisions via `attachmentsMath.ts`'s `resolveCollisionName` (append ` (2)`,
  ` (3)`, etc. before the extension — never overwrites an existing attachment). Returns the
  final name(s) written.
- `remove(name: string): Promise<void>` — deletes one file by name from the attachments folder.
- `thumbnail(name: string): Promise<string | null>` — for image files only (per
  `attachmentsMath.ts`'s `isImageExtension`), reads the file and returns a `data:` URL; `null`
  for non-images or on any read failure. Returns a data URL rather than a raw path because the
  renderer is sandboxed/context-isolated — a `file://` src can't be trusted directly, so the
  main process does the read and hands back inert image data over IPC, the same trust boundary
  every other `aetherElectron` API already uses.
- `open(name: string): Promise<void>` — calls Electron's `shell.openPath` on the file, launching
  whatever the OS normally opens that type with.

### `electron/main.ts`

Four new `ipcMain.handle` registrations — `attachments:list`, `attachments:add`,
`attachments:remove`, `attachments:thumbnail`, `attachments:open` — thin wrappers delegating to
`attachmentsStore.ts`, matching the existing `pty:start` handler's request/response shape (as
opposed to `usage`/`agents`' push-based `:snapshot` events, which fit continuous background
polling — attachments only change on explicit user action, so request/response is the right
shape here, not a subscription).

### `electron/preload.ts`

New `window.aetherElectron.attachments` surface, each method a thin `ipcRenderer.invoke(...)`
call (matching `pty.start`'s existing invoke pattern):

```ts
attachments: {
  list: () => ipcRenderer.invoke('attachments:list'),
  add: () => ipcRenderer.invoke('attachments:add'),
  remove: (name: string) => ipcRenderer.invoke('attachments:remove', name),
  thumbnail: (name: string) => ipcRenderer.invoke('attachments:thumbnail', name),
  open: (name: string) => ipcRenderer.invoke('attachments:open', name),
}
```

### `src/components/files/filesMath.ts` (rewritten in place)

`groupFilesByAgent` is removed (dead once `FilesView.tsx` no longer calls it — see Context).
Replaced with pure, unit-tested logic:

- `resolveCollisionName(existingNames: string[], desiredName: string): string` — returns
  `desiredName` unchanged if not in `existingNames`; otherwise appends ` (2)`, ` (3)`, ... before
  the extension until it finds a name not already present.
- `formatFileSize(bytes: number): string` — human-readable size (`"1.2 KB"`, `"3.4 MB"`, etc.),
  matching this app's existing terse-mono numeric-formatting style elsewhere (e.g. `fmtElapsed`).
- `isImageExtension(name: string): boolean` — checks the name's extension against a fixed set
  (`png`, `jpg`, `jpeg`, `gif`, `webp`, `bmp`).

### `src/components/files/FilesView.tsx` (rewritten)

**No new global store/reducer state.** Attachments are entirely local to this one view —
nothing else in the app reads them — so this component manages its own `useState` list +
loading flag, fetched via `window.aetherElectron.attachments.list()` in a `useEffect` on mount
and re-fetched after every add/remove. This matches the project's precedent of skipping reducer
wiring when a feature is genuinely self-contained (contrast with Uplinks/Settings, which needed
reducer actions because they mutate shared `Cfg`/`providers` read elsewhere).

- Title changes from "FILES" to **"ATTACHMENTS"**.
- A "+ ADD FILE" button (top-right of the card, the slot Terminal's now-removed "SPAWN +" used
  to occupy) calls `attachments.add()`, then re-fetches the list.
- Flat list, most-recently-added first (sorted by `mtimeMs` descending — no grouping; see
  Non-goals). Each row first checks `isImageExtension(name)` (the same pure helper, imported
  directly into the renderer) to decide its own type: image rows show a 32px thumbnail
  (`<img src={dataUrl}>`, fetched lazily per-row via `attachments.thumbnail(name)` — the IPC
  round-trip only happens for rows already known to be images) or a generic file-type badge (reusing this
  view's existing `avatarStyle` idiom, showing the extension in place of an agent's initials)
  for everything else; filename; `formatFileSize(size)`. Clicking a row calls
  `attachments.open(name)`. A small delete (×) affordance per row calls `attachments.remove(name)`
  then re-fetches.
- Empty state: `"no files attached yet — click + ADD FILE to attach a screenshot or document"`.

## State changes

None. No new `AetherState` field, no new reducer action, no `persistence.ts` change — this
slice's data lives entirely on disk (`~/.aether-os/attachments/`) and in `FilesView.tsx`'s own
local component state, not in the app's `localStorage`-persisted reducer state.

## Data flow

1. View mounts → `attachments.list()` → renders rows; image rows independently fetch their own
   thumbnail.
2. "+ ADD FILE" → OS picker → chosen file(s) copied into `~/.aether-os/attachments/` (collision
   handled by `attachmentsMath.ts`) → list re-fetched.
3. Row click → `attachments.open(name)` → OS opens the file in its default app; no in-app state
   change.
4. Delete (×) → `attachments.remove(name)` → list re-fetched.

Nothing outside `FilesView.tsx` reads or is affected by this data — no cross-view wiring, unlike
every prior Phase 3 slice (which each updated exactly one or two other consumers).

## Error handling / edge cases

- **`add()` when the user cancels the picker dialog**: `dialog.showOpenDialog` resolves with
  `canceled: true`; `attachmentsStore.add()` returns an empty array, `FilesView.tsx` treats this
  as a no-op (no re-fetch needed, though a harmless extra fetch is also fine).
- **`thumbnail()` on a corrupt or unreadable image file**: caught in `attachmentsStore.ts`,
  returns `null`; `FilesView.tsx` falls back to the generic file-type badge for that row rather
  than showing a broken image.
- **Name collision on add**: never overwrites — `resolveCollisionName` guarantees a unique name
  before the copy happens.
- **`remove()` on a name that no longer exists** (e.g. removed externally between list-fetch and
  click): the underlying `fs` delete rejects; `FilesView.tsx` catches it, re-fetches the list
  anyway so the UI self-corrects to the actual on-disk state.
- **Attachments folder doesn't exist yet** (first run): `attachmentsStore.ts` creates it
  lazily on first `list()`/`add()` call rather than at app startup — no user-visible difference,
  avoids adding startup work for a feature that may never be used in a given session.

## Testing

**Unit (`attachmentsMath.test.ts`, replacing `filesMath.test.ts`):**
- `resolveCollisionName` returns the input unchanged when not present in `existingNames`.
- `resolveCollisionName` appends ` (2)` on a first collision, ` (3)` if ` (2)` is also taken,
  preserving the original extension (e.g. `screenshot.png` → `screenshot (2).png`).
- `formatFileSize` formats bytes/KB/MB boundaries correctly (e.g. `0`, `999`, `1000`,
  `1_000_000`).
- `isImageExtension` is true for each supported extension (case-insensitive), false for others.

**Not unit tested (per this project's established precedent for `electron/*.ts`):**
`attachmentsStore.ts`'s `fs`/`dialog`/`shell` calls — no automated coverage for fs-touching
main-process code anywhere else in this project either.

**Manual GUI QA (plan-exit, per this project's convention):**
1. Attachments tab (formerly Files) renders the empty state on first use.
2. "+ ADD FILE" opens the native picker; selecting an image adds it, shows its thumbnail, and
   the correct formatted size.
3. Selecting a non-image file adds it with the generic badge instead of a thumbnail.
4. Adding a file with the same name as an existing attachment produces a ` (2)`-suffixed copy,
   not an overwrite — confirm both files are listed and openable independently.
5. Clicking a row opens the file in its OS default application.
6. Delete (×) removes a row and re-fetching confirms it's gone from `~/.aether-os/attachments/`
   on disk.
7. Reload the app — attachments persist (they're real files on disk, not reducer state).
8. Confirm no regressions elsewhere: Chat's fictional-agent file-touched copy
   (`localResponder.ts`/`systemPrompt.ts`) is unaffected — `Agent.files` and its seed data in
   `initialState.ts` are untouched by this slice.
