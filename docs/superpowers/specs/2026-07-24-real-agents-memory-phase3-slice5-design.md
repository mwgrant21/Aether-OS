# Real Dispatch Completions in Memory (Phase 3, slice 5) — Design

## Context

Memory (`src/state/types.ts`'s `MemoryStub`) has a `source: string` field, populated at four call sites:

- `src/state/reducer.ts:83-93` — `applyApprovalResolution`'s HIGH-risk-approval trigger, `source: req.agent`.
- `src/components/terminal/commands.ts:110-126` — the `kill` command's auto-trigger, `source: hit.name`.
- `src/components/terminal/commands.ts:159-169` — the `remember <text>` manual command, `source: 'operator'`.
- `src/components/terminal/commands.ts:217-236` — a second, independent HIGH-risk-approval call site (`commands.ts`'s own `approve`/`deny` cases), `source: req.agent`, matching the first.

Three of the four sites resolve `source` from `state.agents` (the fictional roster) — `req.agent` (`Approval.agent: string`, generated by the fictional simulation's approval-request flow) or `hit.name` (`state.agents.find(...)`). This is the last item from Phase 3's "next open question" (Files and Chat's per-agent channels being the others; Files was resolved separately as a standalone feature, not a Phase 3 slice — see `PROGRESS.md`).

Unlike every prior Phase 3 slice, this one is **not** a relabeling problem. Investigated and confirmed with the user before design: the *triggering events themselves* (an agent being killed, a HIGH-risk approval being resolved) exist only in the fictional simulation (`tick.ts` drives the fictional roster's approval-request generation and lifecycle) — a real Claude Code dispatch is never "killed" or routed through this app's approval queue. There is no real event to re-source these four existing triggers from. The user chose, instead of forcing a relabeling (as Analytics did in slice 4) or redesigning the existing triggers, to **add a new, genuinely real trigger** alongside the untouched fictional ones: auto-create a Memory entry when a real dispatch completes.

`state.realAgents: RealAgentDispatch[]` (from `src/state/liveAgentsMath.ts`, populated every ~1s tick via the `SET_REAL_AGENTS` reducer action) already tracks currently-open real dispatches. A dispatch "completing" is observable as its `toolUseId` disappearing from that list between one tick and the next.

## Goal

When a real dispatch disappears from `state.realAgents` (completes), synthesize a `MemoryStub` for it with `source` set to the dispatch's real `subagentType` — the first Memory-creation trigger in this app driven by real data instead of the fictional roster.

## Non-goals

- **No changes to any of the four existing trigger sites.** `applyApprovalResolution`, `kill`, `remember`, and the second approve/deny call site all keep their current fictional-`source` behavior untouched. This slice is purely additive.
- **No filtering, dedup, or rate-limiting of which completions create a memory.** Every dispatch that disappears from `state.realAgents` creates exactly one `MemoryStub`, matching this app's existing precedent of the HIGH-risk-approval trigger firing unconditionally on every resolution (approve or deny, no filter).
- **No UI changes.** `MemoryRosterCard.tsx`/`MemoryDetailCard.tsx` already render `source` as a free-form string badge with no special-casing per value — a real `subagentType` string renders exactly like a fictional agent name or `'operator'` does today. Confirmed by reading both files: neither branches on `source`'s value.
- **No electron-layer changes.** Detection is a pure renderer-side diff of `state.realAgents` (old) against the incoming `SET_REAL_AGENTS` action's `agents` (new) — `electron/liveAgentTracker.ts`, `electron/main.ts`, `electron/preload.ts`, and `src/aetherElectron.d.ts` are all untouched.
- **Explicit accepted limitation, not solved here:** if the user switches which Claude Code session is globally the most-recently-active (the existing "global, not just this app's own terminal" behavior from Phase 3's first slice) while a dispatch is still genuinely running in the session being switched away from, that dispatch's `toolUseId` will disappear from the next snapshot and this slice will synthesize a "completed" memory for it even though it didn't actually complete — a false positive. This was raised to the user as a real precision-vs-scope trade-off against a heavier electron-side alternative (extending the existing task-notification parser in `liveAgentTracker.ts` to report genuine completions specifically); the user chose the simpler renderer-side diff and accepted this narrow edge case, in the same spirit as this app's other documented session-switch quirks (e.g. slice 1's replay-on-switch behavior).

## Architecture

### `src/state/liveAgentsMath.ts` (new function, existing file)

```ts
export function detectCompletedDispatches(oldAgents: RealAgentDispatch[], newAgents: RealAgentDispatch[]): RealAgentDispatch[] {
  const stillOpen = new Set(newAgents.map((a) => a.toolUseId));
  return oldAgents.filter((a) => !stillOpen.has(a.toolUseId));
}
```

Pure, symmetrical to `applyLinesToOpenDispatches` already in this file — both operate on `RealAgentDispatch[]` by `toolUseId`. Returns the full prior record (all fields) for each dispatch that disappeared, so the reducer has everything it needs (`subagentType`, `description`) without a second lookup.

### `src/state/reducer.ts` (`SET_REAL_AGENTS` case, extended)

Current:

```ts
case 'SET_REAL_AGENTS':
  return { ...state, realAgents: action.agents };
```

Extended to build one `MemoryStub` per completed dispatch, inline — matching the existing HIGH-risk-approval trigger's own inline style a few lines above in the same file (that block was never extracted into `memoryMath.ts`, so this doesn't invent a new pattern):

```ts
case 'SET_REAL_AGENTS': {
  const completed = detectCompletedDispatches(state.realAgents, action.agents);
  let memories = state.memories;
  let memSeq = state.memSeq;
  for (const dispatch of completed) {
    const label = dispatch.description || dispatch.subagentType;
    memories = [
      ...memories,
      {
        id: memSeq,
        name: label,
        content: `${dispatch.subagentType} dispatch completed: ${dispatch.description || 'no description'}`,
        source: dispatch.subagentType,
        ts: nowShort(),
        pinned: false,
        strength: 100,
      },
    ];
    memSeq += 1;
  }
  return { ...state, realAgents: action.agents, memories, memSeq };
}
```

`nowShort` is already imported in `reducer.ts` (`import { nowShort } from '../utils/format';`, used elsewhere in the same file for other memory-creation sites — verified by reading the file). `reducer.ts` already has `import type { RealAgentDispatch } from './liveAgentsMath';` (type-only); `detectCompletedDispatches` needs a separate value import from the same module, e.g. a new `import { detectCompletedDispatches } from './liveAgentsMath';` line alongside it.

## State changes

None beyond the existing `memories`/`memSeq` fields, which are already part of `AetherState` and already whitelisted in `persistence.ts`. No new `AetherState` field, no new reducer action — this extends the payload of the existing `SET_REAL_AGENTS` case.

## Data flow

Every ~1s tick, `useRealAgentsSync.ts` dispatches `SET_REAL_AGENTS` with the latest open-dispatch snapshot (unchanged, existing flow from Phase 3 slice 1). The reducer now additionally diffs the incoming list against the outgoing `state.realAgents` before replacing it; any dispatch that dropped out gets a `MemoryStub` appended to `state.memories`. The Memory view (`MemoryRosterCard`/`MemoryDetailCard`) picks these up exactly like any other memory — no separate code path.

## Error handling / edge cases

- **Multiple dispatches completing in the same tick:** each gets its own `MemoryStub`; `memSeq` increments once per completion within the loop, so no id collisions.
- **A dispatch with an empty `description`** (a real, documented possibility — `liveAgentsMath.ts`'s `applyLinesToOpenDispatches` already defaults it to `''` when absent from the tool-use input): `name`/`content` fall back to `dispatch.subagentType` / `'no description'` respectively rather than rendering an empty-titled memory row.
- **No completions in a given tick** (the common case — most ticks, nothing finished): `detectCompletedDispatches` returns `[]`, the loop doesn't run, `memories`/`memSeq` are reassigned to themselves unchanged (same value, not a new array reference — matches this file's existing no-op convention elsewhere, e.g. `TOGGLE_PROVIDER_CONNECTION` on an unknown name).
- **First tick of a session** (`state.realAgents` starts as `[]`): nothing can "disappear" from an empty list, so no spurious memories on startup.
- **Session-switch false positive:** documented under Non-goals above as an accepted, known limitation — not defended against.

## Testing

**Unit (`liveAgentsMath.test.ts`):**
- `detectCompletedDispatches` returns `[]` when the two lists are identical.
- Returns the one dispatch present in `oldAgents` but absent from `newAgents` (single completion).
- Returns multiple dispatches when several disappear at once.
- Returns `[]` when a dispatch is only *added* (present in `newAgents`, not `oldAgents`) — an addition is not a completion.
- Correctly separates simultaneous add + remove — a dispatch newly appearing and a different one disappearing in the same call.

**Unit (`reducer.test.ts`, `SET_REAL_AGENTS` cases):**
- A dispatch disappearing between the old and new `agents` list creates exactly one `MemoryStub` with `source` equal to that dispatch's `subagentType`, `name`/`content` derived from its `description`, `pinned: false`, `strength: 100`.
- `memSeq` increments by exactly the number of completions when multiple dispatches complete in the same action.
- No new memory is created when `action.agents` contains the same set of `toolUseId`s as `state.realAgents` (nothing completed).
- A completed dispatch with an empty `description` produces `name`/`content` falling back to `subagentType`/`'no description'`.
- Existing `state.memories` entries are preserved (appended to, not replaced) — mirrors the existing HIGH-risk-approval trigger's own test convention.

**Manual GUI QA (plan-exit, per this project's convention):** since triggering a real dispatch to complete organically during a manual QA pass isn't reliably reproducible on demand (matches every prior Phase 3 slice's "no real dispatch was actively running to observe" accepted gap), this slice's manual QA is scoped to confirming no regression in the existing fictional triggers (kill, HIGH-risk approve/deny, `remember`) and confirming the Memory view still renders correctly with a live `electron:dev` session running (even if no completion is captured live during the QA window itself) — the unit tests are the primary correctness evidence for the new code path itself, consistent with this app's stated precedent for `electron/*.ts`-adjacent real-data logic that unit tests already cover thoroughly.
